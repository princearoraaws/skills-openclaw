# Mind Map Generator — 思维导图生成器

## 触发时机

- 用户说"生成思维导图"、"画脑图"、"做一个思维导图"
- 用户总结文章/内容后说"帮我做成思维导图"
- 用户说"把这些内容可视化"、"帮我梳理结构"
- 用户说"导出为 XMind / PNG / PDF"等格式

---

## 思维导图的本质：空间化的论点结构

**思维导图不是大纲的图形版。** 两者本质不同：

| | 大纲/列表 | 思维导图 |
|---|---|---|
| 结构 | 线性、从上到下 | 放射状、从中心向外 |
| 中心节点 | 标题（话题标签） | **核心洞见**（最重要的一句话） |
| 分支 | 章节编号 | **理解该洞见的独立维度** |
| 子节点 | 子章节 | **使分支成立的证据/机制/例子** |
| 读法 | 顺序阅读 | 任意起点，随意跳跃 |
| 价值 | 存储信息 | **揭示结构、激发联想** |

优秀的思维导图让人一眼看出：**核心是什么、由哪几个维度支撑、每个维度的关键证据是什么。**

---

## 第一步：确定中心节点（最重要）

中心节点决定整张图的质量。

**错误示范（话题标签）：**
- ❌ "AI 发展趋势"
- ❌ "产品发布计划"
- ❌ "Python 学习"

**正确示范（核心洞见）：**
- ✅ "AI 正从工具变为协作者"
- ✅ "产品成功 = 用户价值 × 执行速度"
- ✅ "Python 的价值在于生态而非语法"

**判断标准：** 如果中心节点换成另一个话题，整张图的分支还能成立吗？能成立说明中心节点太泛，需要更精确。

**确定方法：** 问自己"关于这个内容，最重要的一件事是什么？" 答案就是中心节点。

---

## 第二步：设计主分支（维度而非章节）

主分支是"理解中心洞见的独立视角"，不是原文的章节目录。

### 分支设计原则

**原则 1：分支之间必须相互独立**

每个分支回答关于中心节点的一个不同问题。常用的维度框架：

| 内容类型 | 分支维度框架 |
|----------|-------------|
| 分析/研究类 | 是什么 · 为什么 · 怎么做 · 结果如何 |
| 问题解决类 | 根因 · 影响 · 方案 · 评估 |
| 学习知识类 | 核心概念 · 运作机制 · 适用场景 · 常见误区 |
| 产品/项目类 | 目标 · 策略 · 执行 · 风险 |
| 比较分析类 | 相同点 · 差异点 · 各自优势 · 选择建议 |
| 叙事/报告类 | 背景 · 发现 · 影响 · 行动 |

**注意：** 维度框架是引导，不是模板。根据内容实际情况选择最能揭示结构的维度。

**原则 2：所有分支必须共同服务于中心节点**

每个分支都应该是"中心节点成立"的一个支柱。反问："去掉这个分支，对理解中心节点有损失吗？"没有损失则该分支是冗余的。

**原则 3：分支数量 3–6 个，不以"原文有几节"为准**

- 少于 3 个：维度不足，理解片面
- 超过 7 个：维度太碎，失去聚焦
- 不要因为原文有 8 章就设 8 个分支，合并同类维度

**原则 4：同级分支的抽象层次必须一致**

❌ 错误：同级出现"技术原理"（抽象）和"GPT-4"（具体实例）
✅ 正确：同级都是维度（"技术路线"、"商业模式"、"社会影响"）

---

## 第三步：填充子节点（证据而非子话题）

子节点不是"这个分支下还有哪些话题"，而是"**让这个分支成立的具体依据**"。

### 子节点质量标准

**能让分支"落地"的内容才是好子节点：**

| 子节点类型 | 示例 | 作用 |
|------------|------|------|
| 具体机制 | "注意力机制使模型聚焦关键词" | 解释"为什么" |
| 量化数据 | "GPT-4 训练成本超 1 亿美元" | 使声明可信 |
| 典型案例 | "Copilot 使代码效率提升 55%" | 使抽象具体 |
| 对比参照 | "vs 传统搜索：主动生成 vs 被动检索" | 揭示差异 |
| 行动指南 | "先用小数据集验证再扩规模" | 指导实践 |
| 关键限制 | "幻觉率随任务复杂度非线性上升" | 防止误用 |

**叶节点（最深层）必须是具体的：**
- ❌ "性能问题" → ❌ "优化" → ❌ "改进方向"（三层都是空洞标签）
- ✅ "推理延迟高" → ✅ "量化压缩" → ✅ "INT8 精度损失 < 1%"（越深越具体）

### 层级深度规则

```
中心节点（1 个）
  └── 主分支（3–6 个）    ← 理解维度，抽象，词组
        └── 子节点（2–5 个）  ← 支撑证据，半具体，词组或短句
              └── 叶节点（可选，1–3 个）  ← 最具体的事实/数据/步骤
```

- 每个主分支至少有 2 个子节点（1 个子节点说明该维度未被充分挖掘）
- 叶节点不超过 3 个（太多说明该子节点还可以再拆一层）
- 全图最多 4 层（超过 4 层说明某个维度的粒度失控了）

---

## 第四步：平衡性检查（视觉即信息）

思维导图中，**视觉上的不平衡本身就是信息**。生成完 JSON 后：

**检查 1：分支间子节点数量差距**

某分支有 8 个子节点，其他只有 2 个 → 要么那个分支内容确实更复杂（可以接受，但需确认），要么其他分支挖掘不够（需要补充）。

**检查 2：所有分支是否都对等重要**

如果某个分支删掉对理解中心节点影响很小，说明它是凑数的维度，应当合并或删除。

**检查 3：叶节点具体程度是否一致**

有的叶节点是具体数据，有的是模糊标签，说明处理深度不一致——补充模糊的那些。

---

## 第五步：节点文字规范

| 规则 | 正确 | 错误 |
|------|------|------|
| 使用原文关键词，不过度意译 | "联邦学习" | "一种保护数据的分布式方法" |
| 动词短语比名词堆砌更有力 | "降低推理延迟 40%" | "推理延迟优化相关内容" |
| 长度 4–12 字 | "本地训练不传原始数据" | "关于在本地进行模型训练而不需要传输原始用户数据的技术方案" |
| 并列内容拆成独立节点 | 三个节点："医疗" / "教育" / "金融" | 一个节点："医疗教育金融" |
| 越深层越具体 | 叶节点写具体数字/步骤/案例 | 叶节点写抽象标签 |

---

## 第六步：构造 JSON 并执行

```json
{
  "central": "核心洞见（不超过 15 字）",
  "branches": [
    {
      "label": "维度1",
      "color": "#4A90D9",
      "children": [
        "具体证据A",
        "具体证据B",
        {
          "label": "需要进一步展开的证据C",
          "children": ["最具体的事实1", "最具体的事实2"]
        }
      ]
    }
  ]
}
```

**颜色分配建议（按分支语义）：**

| 颜色 | 适用维度 |
|------|----------|
| `#4A90D9` 蓝 | 机制/原理/方法论 |
| `#27AE60` 绿 | 成果/价值/应用 |
| `#E86C3A` 橙 | 问题/挑战/风险 |
| `#9B59B6` 紫 | 背景/趋势/上下文 |
| `#F39C12` 黄 | 资源/工具/要素 |
| `#1ABC9C` 青 | 流程/步骤/路径 |
| `#E74C3C` 红 | 警告/限制/禁忌 |

脚本路径：`{baseDir}/generate_mindmap.py`

```bash
python3 {baseDir}/generate_mindmap.py \
  --title "标题" \
  --output ~/Desktop/mindmap.html \
  --data '{ ... }'
```

参数：`--format html/png/jpg/svg/pdf/xmind`，`--output` 可省略，`--scale` 控制 PNG 清晰度（默认 2.0）。

## 完整示例：从文章到高质量思维导图

**输入：** 一篇分析大模型商业化路径的文章，讲了技术门槛、盈利模式、竞争格局、监管风险四个方面。

**第一步：确定中心节点**
- 不是 ❌ "大模型商业化"（话题标签）
- 是 ✅ "大模型商业化：技术壁垒正被成本竞争取代"（核心洞见）

**第二步：设计分支维度**
文章四个方面（技术/盈利/竞争/监管）本身已是独立维度，保留；但"竞争格局"可以拆成"竞争驱动力"和"竞争结果"两个维度，更有分析价值。

**第三步：填充子节点**
不是把段落摘要放进去，而是找"使该分支成立的具体证据"：

        ]
      },

        "label": "主流盈利路径",

        "children": [
          "API 按 token 计费",
          {"label": "垂直行业定制", "children": ["医疗合规要求高溢价", "法律文档审核替代人工"]},
          "订阅制 Pro 用户"
        ]
      },
      {
        "label": "竞争驱动力",
        "color": "#E86C3A",
        "children": [

## 常见质量问题自查

生成 JSON 后，对照以下问题检查，有问题立即修改：

**中心节点：**
- [ ] 把中心节点换成另一个话题，现有分支还能成立吗？（能成立 → 中心节点太泛，需要更精确）
- [ ] 中心节点是一个名词短语，还是一个有观点的陈述？（名词短语 → 尝试找到核心洞见）

**主分支：**
- [ ] 所有分支是否在同一抽象层次？（有具体实例混入 → 降级为子节点）
- [ ] 每个分支是否都在回答关于中心节点的不同问题？（重复或包含关系 → 合并）
- [ ] 分支数量是否在 3–6 个之间？

**子节点：**
- [ ] 每个子节点是否使其父分支"更可信、更具体、更可操作"？（否 → 重新找证据）
- [ ] 最深层的叶节点是否是具体的事实、数据或步骤？（是模糊标签 → 替换为具体内容）
- [ ] 是否有分支只有 1 个子节点？（1 个 → 该维度没挖掘够，或该子节点应升级为分支）

**整体：**
- [ ] 视觉上最大的分支，其内容重要性是否真的高于其他分支？（否 → 压缩该分支或补充其他）
- [ ] 删掉任意一个主分支，对理解中心节点的损失是否明显？（损失不明显 → 该分支是冗余的）

## 首次安装依赖

HTML / SVG / XMind **无需依赖**，开箱即用。PNG / JPG / PDF 需要 `cairosvg`。

```bash
# macOS
brew install cairo pango libffi && pip3 install cairosvg pillow

# Ubuntu / Debian
sudo apt-get install -y libcairo2-dev libpango1.0-dev libgdk-pixbuf2.0-dev libffi-dev
pip install cairosvg pillow --break-system-packages

# Fedora / RHEL
sudo dnf install cairo-devel pango-devel libffi-devel
pip install cairosvg pillow --break-system-packages
```
