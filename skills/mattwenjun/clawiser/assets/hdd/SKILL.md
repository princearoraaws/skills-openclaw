# HDD: Hypothesis-Driven Development

七步流程：**Step 0 → HT → PT → AT Design → DEV → AT Run → E2E**

每个测试阶段的核心动作相同：**写测试用例 → 执行 → 判定通过/不通过**。
信息收集（查数据、搜网、读文档、跑命令）是写测试用例的输入，不是判定依据。

## 流程

### 0. Resource Mapping（资源映射）— 动手前必做

**在提假设之前，先回答："我有哪些资源可以帮我解决这个问题？"**

```
- 本地资源:
  - HANDOFF / product-plan: 有没有已规划但未实现的方案？
  - 已有代码/配置: 有没有相关但未启用的能力？
  - memory_search: 之前讨论过这个问题吗？
- 外部资源:
  - 官方文档: 涉及的 API/工具有没有文档？先读完再动手
  - 社区/搜索: 有没有已知的最佳实践？
- 平台资源:
  - OpenClaw 工具: gateway/cron/sessions 等工具能直接解决吗？
  - 配置能力: 有没有现成的配置项可以调？
```

**为什么是 Step 0：** 如果跳过资源映射直接提假设，会在错误的解空间里浪费大量时间。已有的文档、API key、配置项、工具能力——任何一条都可能在几分钟内解决问题，但没做资源映射就看不到它们。

### 1. HT: Hypothesis Test（假设验证）

提出假设，用测试用例验证。

```
- 假设: [对问题根因的判断]
- 信息收集: [查本地文件/搜网/跑命令 — 为写测试服务]
- 测试用例:
  - HT1: [验证条件] → 预期: [xxx] → 实际: [xxx] → ✅/❌
  - HT2: ...
- 判定: ✅ 进入 PT / ❌ 替换假设，重跑 HT
```

**关键规则：**
- 不能提了假设就一直绕，必须先用测试用例确认假设是否成立
- 假设不通过 → 必须替换假设，不能在错误假设上调方案

**假设提出后必须自我攻击：**
1. **攻击 Ground Truth**：我的判断依据本身有没有破绽？有没有反例？
2. **检查其他可解释性**：同一现象是否有完全不同的解释？
3. **找反例**：已有事实中，有没有直接反驳假设的证据？

三项检查不通过 → 假设不成立，重新提假设。

### 2. PT: Plan Test（方案验证）

提出方案，用测试用例验证可行性。

```
- 方案: [具体方案描述]
- 信息收集: [查文档/搜网/确认可行性 — 为写测试服务]
- 测试用例:
  - PT1: [方案前提是否成立] → 预期: [xxx] → 实际: [xxx] → ✅/❌
  - PT2: [关键参数是否可用] → 预期: [xxx] → 实际: [xxx] → ✅/❌
- 判定: ✅ 进入 AT Design / ❌ 调整方案 / ❌❌ 回退 HT
```

**回退规则：**
- PT 失败但假设没问题 → 调整方案，重跑 PT
- PT 失败且发现假设有问题 → 回退 HT，重新提假设

### 3. AT Design: Acceptance Test Design（验收用例编写）

在写代码之前，先定义"做完长什么样"。

```
- 验收标准:
  - AT1: [输入/操作] → 预期结果: [xxx]
  - AT2: [边界条件] → 预期结果: [xxx]
```

必须在 DEV 之前完成。用例要具体可执行，不能是模糊描述。

### 4. DEV: Development（开发）

基于 PT 方案 + AT 用例执行开发。

```
- 改动清单:
  - [文件1]: [改了什么]
  - [文件2]: [改了什么]
```

### 5. AT Run: Acceptance Test Run（验收测试执行）

执行 AT Design 中定义的用例。

```
- AT1: → 实际: [xxx] → ✅/❌
- AT2: → 实际: [xxx] → ✅/❌
- 判定: ✅ 进入 E2E / ❌ 回 DEV 修复
```

### 6. E2E: End-to-End Test（端到端验收）

由用户亲自验收。

```
- E1: [验收项] → ✅/❌
- 判定: ✅ 完成 / ❌ 回退 DEV 或 PT
```

## 回退路径

```
Step 0 (Resource Mapping)
 ↓
HT ←────────────────────┐
 ↓ pass                  │
PT ←─────┐               │
 ↓ pass  │ PT fail       │ 根因在假设
AT Design│               │
 ↓       └→ 调整方案 ────┘
DEV
 ↓
AT Run ──→ fail → 回 DEV
 ↓ pass
E2E ────→ fail → 回 DEV 或 PT
 ↓ pass
Done ✅
```

### 收尾检查（E2E 通过后必做）

```
- [ ] 是否需要新建/更新项目文档？
- [ ] 是否有新的经验教训需要记录？
- [ ] 相关的 memory 文件是否已更新？
```

## 何时使用

**需要 HDD（满足任一）：**
- 涉及"为什么"（诊断/debug）
- 涉及"哪种方案更好"（优化/选型）
- 改动影响上下游（架构/配置变更）
- 结果不确定，需要验证

**不需要 HDD：**
- 指令明确、结果确定（"把标题改成xxx"）
- 纯信息查询（"这个 API 怎么用"）
- 简单追加/删除操作

## 异常模式识别

HT 阶段之前，还有一个更根本的能力：**从数据中发现异常**。

### 看整体模式，不只看单条数据

- **不自然的一致性**：分类器把所有输入归为同一类 → 输入可能是错的
- **不该出现的空值**：某个字段全是 null → 上游数据没传进来
- **计数不对**：预期 8 种结果但只看到 1 种 → 系统性问题

### 保持对"正常"的校准

1. 主动检查几条典型输入的处理结果，建立基线
2. 对关键环节保留日志/统计
3. 定期抽查，不只是出了问题才看

## 调试技术

### 逐步打桩法

多步处理链输出不对时：
1. 每步之间插入 debug log（输入和输出）
2. 对比每步结果，定位第一个"输入对但输出错"的步骤
3. 问题就在那一步

**不要猜。** 打桩定位比猜测修复快得多。

### 模拟 vs 真实验证

- 本地模拟可以快速验证逻辑
- **模拟通过 ≠ 真实通过**——模拟的输入格式可能与真实环境不同
- 必须在真实环境再验证一次

### 格式假设检查

当"逻辑对但不工作"时：
1. `JSON.stringify()` 打出原始文本 → 暴露不可见字符
2. `.length` 确认完整性 → 排除截断
3. 注意：**单行 vs 多行**、`\n` vs `\r\n`

### 减少调试来回

1. 每次重启前多加几个 debug 点，一次收集尽量多信息
2. 同时打印输入和输出
3. 保留 debug log 到问题完全解决
