# SDD — 场景驱动开发（Scenario-Driven Development）

**先回答"谁在什么场景下怎么用"，再回答"怎么实现"。**

常见错误：收到需求 → 直接开干 / 做技术方案 / 列功能清单。
正确顺序：收到需求 → **想清楚场景 → 定义判断依据 → 分步计划** → 然后才动手。

不管你做的是不是"产品"，这个思路都管用。写论文要先想清楚给谁看、论点是什么；做学习计划要先想清楚目标是什么、怎么判断有没有效；做方案要先想清楚对方在什么处境下看到这个方案。

## 三要素

### 1. 场景（必须先想清楚）

用"一天的使用故事"描述，不用功能列表：

```markdown
## 使用场景

### 场景 1：每日推送
用户早上打开消息，看到今天的精选内容。
5 条带一句话理由，点开链接直达原文。
不需要主动操作，推送自动到。

### 场景 2：深度展开
看到一条感兴趣的，回复"展开第 3 条"。
收到核心段落 + 相关引用 + 可能的 idea 候选。
```

**检验标准**：读完场景描述，一个不了解这件事的人能想象出画面吗？能 → 够了。不能 → 写得不够具体。

### 2. 判断依据（什么算"好"）

每个场景定义**可验证的质量标准**：

```markdown
## 判断依据

| 维度 | 标准 | 怎么验 |
|------|------|--------|
| 相关性 | 推送内容与兴趣匹配度 ≥70% | 用户反馈 |
| 时效性 | 80% 是最近 3 天的 | 查日期分布 |
| 可操作 | 每条都能一键展开 | 功能测试 |
```

### 3. 分步计划（怎么一步步做到）

基于场景**反推**步骤：
- 第一步 = 最小可用（覆盖场景 1）
- 第二步 = 核心体验（覆盖场景 2）
- 后续 = 增强和优化

**不是按技术模块拆步骤，是按用户价值拆。**

## 常见陷阱

| 陷阱 | 表现 | 纠正 |
|------|------|------|
| 技术驱动 | "先做抓取，再做评分，再做推送" | 倒过来：推送什么样 → 需要什么评分 → 需要什么数据 |
| 功能堆砌 | 第一步就塞满功能 | 砍到只剩场景 1 能跑通的最小集 |
| 跳过场景 | 直接写任务列表 | 先写"用户每天看到什么" |
| 假设未验 | "用 XX API 做搜索" → 没查过能不能用 | 每个关键假设走 HDD 验证 |

## 产出物

完成 SDD 后，应产生：

1. **product-plan.md**（存 `memory/projects/<project>/`）
   - 使用场景（至少 2 个核心场景）
   - 判断依据（每个场景的质量标准）
   - 分步计划（按用户价值排序）
2. **第一步详细计划**
   - 任务列表 + 每个任务的验收标准
   - 依赖和阻塞项

技术选型、架构设计在场景理清楚之后再做。
