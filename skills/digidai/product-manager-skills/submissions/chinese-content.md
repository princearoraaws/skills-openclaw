## 1. 人人都是产品经理 (woshipm.com)

**标题：我给 Claude Code 装了一个产品经理大脑：6 个知识域、32 个 SaaS 指标、30+ 框架**

---

去年我开始用 AI 辅助产品工作，从写 PRD 到分析指标，从竞品研究到用户访谈准备。用了一圈下来，有一个感受越来越强烈：通用 AI 做产品经理的活，总差那么一口气。

### 通用 AI 做 PM 工作的三个硬伤

**第一，它不会推回来。** 你说「我要做一个 dashboard」，它就老老实实帮你设计 dashboard。不会问你：「等等，你要解决的问题到底是什么？这是不是 Solution Smuggling？」一个合格的产品经理应该在 framing 阶段就发现问题，而不是当一个需求翻译机。

**第二，它给的指标建议是空话。** 问它「怎么降低流失率」，它会说「优化用户体验、提升产品价值」。这种回答等于没说。一个真正懂 SaaS 的 PM 会告诉你：你的月流失率 8% 意味着年流失率 63%（公式：`1 - (1-0.08)^12`），这已经是红色警报，然后给你分析是激活问题、价值交付问题还是定价问题。

**第三，它把所有框架一视同仁。** RICE、ICE、Kano、加权评分——它能把每个框架解释得头头是道，但不会告诉你：你现在这个阶段，用 RICE 其实是错的，你应该先做 Cost of Delay 分析。框架不是越多越好，而是要在对的时机用对的那个。

### 所以我做了一个「PM 大脑」

这不是一个模板包，也不是一个 prompt 集合。它是一个完整的产品经理知识系统，安装到 Claude Code 之后，AI 就具备了一个资深 PM 的领域认知。

整个系统分为 6 个知识域：

| 知识域 | 核心能力 | 代表框架 |
|--------|---------|---------|
| 发现与研究 | 验证问题、用户访谈、机会识别 | JTBD、Mom Test、机会解决方案树 |
| 战略与定位 | 产品定位、市场分析、路线规划 | Geoffrey Moore 定位、PESTEL、TAM/SAM/SOM |
| 交付物与执行 | PRD、用户故事、史诗拆分 | Cohn 用户故事、8 种拆分模式、PRFAQ |
| 财务与指标 | 32 个 SaaS 指标计算与诊断 | MRR/ARR/NRR、CAC/LTV、Rule of 40 |
| 职业与领导力 | PM 到总监到 VP 的晋升路径 | 海拔-视野框架、三 P 框架 |
| AI 产品 | AI 原生产品的构建方法论 | AI 就绪评估、上下文工程、Agent 编排 |

每个知识域都是按需加载的——你问 PRD 相关的问题，它只加载「交付物与执行」模块，不会把 13 万字的知识全部塞进上下文。

### 深入看：SaaS 指标模块

这是我花最多时间打磨的部分。32 个指标，每一个都包含：

- **精确公式**（不是概念描述，是可直接计算的公式）
- **阶段基准值**（早期/成长期/规模化，各阶段的健康范围不同）
- **红旗严重等级**（什么程度该警觉，什么程度该拉警报）

你可以直接说：「分析一下这些指标：MRR 5 万美元，月流失 8%，CAC 500 美元，LTV 3000 美元」，它会给你一个完整的业务健康诊断，包含严重等级和行动建议。

不是「你的 CAC 有点高」这种废话，而是「你的 LTV:CAC 比率是 6:1，表面看不错，但结合 8% 月流失率，你的 Payback Period 超过 12 个月，在现有留存水平下 LTV 被严重高估，实际 LTV:CAC 可能不足 3:1」。

### 反模式检测：它会主动找茬

这个系统内建了 40 多种 PM 反模式检测，覆盖所有 6 个知识域：

- **Solution Smuggling**（解决方案走私）：需求描述里偷偷塞了方案
- **Metrics Theater**（指标表演）：追踪好看但不驱动决策的指标
- **Feature Factory**（功能工厂）：没验证问题就开始做功能
- **HiPPO 优先级**：路线图被最高薪那位的意见左右
- **过早规模化**：单位经济模型还没跑通就开始猛推增长

当你写 PRD 时说「用户需要一个推荐系统」，它会标记这是 Solution Smuggling，追问你：「用户遇到的问题是什么？是发现效率低、选择疲劳、还是相关性不足？」

## 3. 知乎回答

**问题：产品经理日常使用的 AI 工具有哪些？**

---

说几个我实际在用的，不搞大而全的盘点。

**日常高频：**

1. **ChatGPT / Claude** — 竞品调研、方案脑暴、文案润色。大家都在用，不展开了。
2. **Cursor / Claude Code** — 写需求文档时直接在 IDE 里跟 AI 协作，尤其适合技术型 PM。
3. **Notion AI** — 会议纪要整理、文档摘要，嵌在工作流里比较自然。
4. **Miro AI** — 头脑风暴和用户旅程图的辅助生成，团队协作场景。

但说实话，用了一段时间之后，你会发现一个问题：**这些工具都是通用型的，它们不具备产品经理的领域知识。**

举个例子。你让 ChatGPT「帮我分析一下我的 SaaS 业务健康度」，给它月流失率 8%。它会说「8% 的月流失率偏高，建议优化用户体验」。

但一个有经验的 PM 会怎么回应？先算：月流失 8% 意味着年流失 `1-(1-0.08)^12 = 63%`。然后判断：这是红色警报。接着追问：你的 CAC 回收周期多久？NRR 多少？有没有扩展收入在对冲？最后给出分阶段的止血方案。

这不是 AI 能力不够，而是它缺乏结构化的领域知识。

所以我做了一件事：**给 Claude Code 装了一个 PM 大脑。**

这是一个开源的技能系统（不是 plugin，是纯 Markdown 知识文件），包含：

- **6 个知识域**：发现研究、战略定位、交付执行、财务指标、职业发展、AI 产品
- **30+ 框架**：JTBD、Mom Test、Geoffrey Moore 定位、机会解决方案树、PRFAQ……不是罗列定义，而是带决策逻辑和适用场景的应用指南
- **32 个 SaaS 指标**：每个都有精确公式、分阶段基准值、红旗等级
- **40+ 反模式检测**：自动识别 Solution Smuggling（需求里偷塞方案）、Feature Factory（没验证问题就堆功能）等常见陷阱

装上之后，AI 的回答质量有质的变化。它不再是一个什么都同意的 yes-machine，而是一个会推回来的 PM agent。你说「我们需要做一个 dashboard」，它会问你：「你要解决什么问题？是数据可见性不足还是决策效率低？这看起来像 Solution Smuggling。」

安装就一行命令：

```bash
clawhub install product-manager-skills
```

免费开源，GitHub 地址：https://github.com/Digidai/product-manager-skills

和市面上的 AI PM 工具比，它的差异化很简单：**它不是帮你更快地填模板，而是帮你更好地思考。** 它有观点、会质疑、能计算、知道什么阶段该用什么框架。

当然它也有局限——它不能替你做用户访谈，也不能帮你搞定跨部门政治。但作为一个思考伙伴，尤其是在写 PRD、做指标诊断、准备战略评审的时候，它确实帮我节省了大量时间，而且输出质量比我一个人闷头写要好。
